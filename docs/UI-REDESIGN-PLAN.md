# Voxi UI Redesign — Implementation Plan

> Status: proposed. Synthesizes four design lenses (Shazam-fidelity, brand-purity, motion-loading, engineering) into one buildable spec.
> Scope: UI-only. No changes to the BFF, the NDJSON/events contract, the api client, metering, or ACLs.
> Master specs this defers to: `docs/PLAN.md`, `design.md` (palette/type/shape source of truth), `docs/IMPLEMENTATION-STATUS.md`.

## 1. Context

Voxi is a "real-world Pokédex": the user photographs a human-made object and the Guide — a dry, witty British narrator embodied by an animated Orb — identifies it as specifically as possible, narrates it, can generate a two-voice podcast, and holds a voice/text conversation. One photo equals one durable thread; revisiting resumes it.

**The problem.** The current shell is built around a charcoal ("dark") default surface and a three-tab bottom bar (Capture / Collection / Settings) in `app/app/(tabs)/_layout.tsx`. That chrome buries the product's single hero interaction (point → identify) behind equal-weight tabs, defaults to a cold shell that fights `design.md`'s warm-cream identity, and offers almost no motion or loading choreography — undercutting the "living Guide" feeling the Orb is meant to sell.

**The directive (non-negotiable).** Adopt Shazam's *layout and interaction model* — one big central tappable orb on a full-bleed canvas, concentric pulsing rings while it "searches," a full-bleed result with a bottom-left title and a pill CTA, and a left push-drawer for everything secondary — but render it **entirely** in Voxi's warm-cream system from `design.md` (cream `#F4F1E8` + green `#29AB60`), **never** Shazam's blue, gradients, or glows. Collapse the app to two primary screens (Login, Steady-state) and hide Collection / Settings / Upgrade behind a hamburger-opened left drawer that pushes the main screen right and dims it. Rich animation, transitions, and loading states are a hard requirement.

**Intended outcome.** A cream, minimal-chrome capture home whose center of gravity is a single green capture orb; a processing screen that radiates staged pulsing rings choreographed to real stream events; a full-bleed reveal that reads like a museum placard; and a push-drawer that carries all secondary navigation — all while preserving every `testID`, every E2E assertion, the honesty gate, and the `capture → createThread → stream → reveal` data flow exactly.

## 2. Design direction

**The synthesis in a paragraph.** We keep Shazam's *spatial and motion blueprint* — full-bleed canvas, one central tappable orb, instruction text above it, a "Recently Found" carousel peeking at the bottom, concentric pulsing rings + staged status text during the search, a full-bleed result with bottom-left title and a primary pill CTA, and a left push-drawer where content slides right and dims — and dress every pixel of it in `design.md`'s warm-cream system. The canvas is cream `#F4F1E8` everywhere (the live viewfinder *is* the "full-bleed image"; its web fallback is a flat cream branded canvas — never charcoal, never a gradient wash). The central capture control is a **flat green `#29AB60` disc** (the audio-first PRIMARY lane), carrying only the single shallow card shadow — explicitly **not** Shazam's blue gradient bloom and **not** the violet→cyan→gold `orbGradient`, which stays reserved for the *narrator* Orb on processing and reveal. Serif (Fraunces) appears **only** as the left-aligned `voxi` wordmark; every title, instruction, and status line is Nunito sans. Green is the primary/audio lane, blue `#3D89F5` is links/secondary, terracotta `#C56A3E` is the one warm-exit accent — the lanes never blur. Depth is shallow: one shadow token, opacity-faded hairlines for rings, no glows or halos.

**Five off-brand temptations we explicitly refuse** (from the brand-purity lens): (1) Shazam's dark canvas → keep cream; (2) the blue gradient orb → flat green, no gradient on the *capture* control; (3) glow/bloom halos → opacity-faded hairlines + the single shallow shadow only; (4) cold `#FFFFFF` background or pure `#000` text → white is float-cards only, text is warm `#262524`; (5) a serif result title or a second accent hue → titles are Nunito sans; palette is green + blue + warm neutrals.

**Token reality — three tokens the table references do NOT exist in `theme.ts` today and MUST be added first (§5, sequencing step 1) so no value is invented ad hoc:** `surface.textTertiary` (`parchment.textTertiary '#A3A19B'`, `dark.textTertiary '#5E6061'`), a top-level `scrim` (`rgba(20,18,14,0.35)`), and a `shadow` object token. Reference these token *names*, never raw hex or the module-private `palette`. Extend `theme.test.ts`'s token list + AA sweep to cover them.

**Type scale — use the ready `typeStyles` ramp in `src/lib/fonts.ts`, NOT the coarse `theme.ts` `size` scale.** `size.xl` is **26**, which is NOT design.md's heading 24 — do not use it for the reveal/hero title (that would ship a +2px drift). Titles = `typeStyles.heading` (24), CTA/status = `typeStyles.headline` (17), card title = `typeStyles.name` (16), timestamp = `typeStyles.caption` (12), microcopy = `typeStyles.footnote` (13), stage eyebrow = `typeStyles.overline`.

**Text-tertiary is decorative-only.** `#A3A19B` on cream measures ~2.3:1 — **below AA**. Any *essential* copy (login consent/age-gate/privacy microcopy, error copy, status lines, chip labels) renders in `text-secondary` `#605E58` (~5.7:1), never tertiary. Reserve `text-tertiary` strictly for genuinely decorative, duplicated meta (e.g. a timestamp that is not the only signal). Add tertiary-on-cream to the `theme.test.ts` AA sweep so essential-copy misuse is caught.

**Glow carve-out.** The "no glows/halos" rule has exactly one sanctioned exception: the *narrator* `Orb`'s bloom (`orbGradient`, opacity ~0.25 — "outside design.md scope" per `theme.ts`). The *capture* orb stays strictly flat green with **no** bloom — only the shallow `shadow` token. Do not strip the narrator bloom to "comply," and do not copy a bloom onto the capture orb.

**Chrome over the native viewfinder (token/reality reconciliation).** "App canvas → `parchment.bg`" applies to the **web cream fallback + processing + reveal + login** — NOT the native camera background, which is the live full-bleed viewfinder. Warm-neutral chrome tokens (`text-primary`/`text-secondary`, the shallow card shadow) are illegible over an arbitrary bright photo. On native camera, place a top cream→transparent scrim `View` (mirror the reveal bottom-scrim pattern, ~0→0.45 cream) behind the hamburger + wordmark + instruction, and flip that over-feed chrome/instruction text to `onAccent`/white; add a hairline or scrim strip behind the Recently-Found cards over a live feed. The cream token mapping holds only on the web fallback and the non-camera screens.

## 3. Screen-by-screen spec

Global chrome is **route-scoped, not uniform** (see §3.5 and the §5 `AppHeader` route-gate). On **`camera` only**: transparent borderless bar, `space.lg` (16) gutter, a top-left hamburger (`nav.menuButton`, line icon, 44pt target, `text-primary`) with the left-aligned serif `voxi` wordmark beside it (plus the small docked narrator Orb). On the **pushed** `threads` / `settings` routes there is **no hamburger and no wordmark** — each renders its own top-left 44pt `nav.close` back-chevron → camera (design.md pushed/modal chrome; §3.5). The bottom tab bar is deleted; the drawer (reachable only from camera) re-hosts `nav.threadsTab` / `nav.settingsTab`.

### 3.0 First-run ("A brief hello") — `app/app/first-run.tsx`

Login routes here on success (not straight to camera), so this 4-step persona intro is the user's **first meeting with the Guide** and the single most important "living Guide" beat — it is promoted out of the §5 light-touch list into its own mini-spec.

**Layout & choreography.** Cream canvas, shared header chrome. The narrator `Orb` enters larger (~140) with a one-time settle-in bloom on mount ("meet") and breathes throughout, shifting state per step: `idle` on the meet step → `listening` on the mic/permission step → `thinking` on grant. Each step's copy (sans `heading` title + `body` line, green primary CTA) **FadeRises in as the prior rises out** (keyed `FadeRise`, base 240). On finish, carry the Orb forward as a pseudo-shared-element (like the capture hand-off, §4 row 7) into the capture home so the character feels continuous, not reset.

**Reduce-motion.** Cross-fades between steps; static Orb bloom (no mount bloom animation); Orb never hidden.

**ids.** All `firstRun.*` ids unchanged.

### 3.1 Login — `app/app/welcome.tsx` (id root `welcome.screen`)

**Layout (top → bottom).** (1) **Hero third** — the narrator Orb (`Orb`, size ~120, state `idle`; the one warm-gradient element permitted on this screen) breathing over cream. (2) **Brand block** — serif `voxi` wordmark + a sans `heading` "The Guide" and one `body` line: *"Photograph anything human-made. I'll tell you what it is."* (3) **Field stack** on `surface-sunken` pill fields: `welcome.emailInput`, then `welcome.otpInput` on phase 2; two `green-soft` toggles `welcome.eulaAccept` and `welcome.ageConfirm`; privacy microcopy in `footnote` / `text-tertiary`. (4) **Primary CTA** — the 52pt green `welcome.continueBtn` pinned low.

**Copy.** Phase 1 CTA: "Continue". Phase 2 helper: "We sent a code to {email}." CTA: "Verify". Gate microcopy: "By continuing you agree to the Terms and confirm you're 16 or older."

**States.** default → email entered → `busy` (sending) → phase 2 (OTP) → `busy` (verifying) → success (route to **`/first-run`** — the real post-auth destination in `welcome.tsx`, NOT `/(tabs)/camera`; see §3.0) → error → **offline**.
- **Error placement (attached, not floating).** Render the recoverable `ErrorState` (warm-bordered white `surface` card, in-persona copy, NOT red) *immediately beneath the field it refers to* — the email field in phase 1, `welcome.otpInput` in phase 2 — so it reads as attached to the input, never floating near the low-pinned CTA.
- **Offline.** Pin `global.offlineBanner` at the top of the cream login; keep `welcome.continueBtn` disabled with the microcopy *"You're offline — signing in needs a connection. I'll wait."* directly above the pinned CTA.
All existing `welcome.*` logic and ids unchanged.

**Keyboard-up hierarchy (collapse order).** Wrap the stack in `KeyboardAvoidingView` + a scroll container. On field focus (or phase 2), the 120px hero Orb is the **first thing sacrificed** — shrink it 120→56 and hide the tagline — so the active field, the two gate toggles (`welcome.eulaAccept`/`welcome.ageConfirm`, which the auth gate depends on), and the primary CTA all stay in view above the keyboard. The green CTA is **never** sacrificed. Give the focused-state and reduce-motion layouts explicitly (reduce-motion: instant resize, no orb animation).

**Form accessibility.** `welcome.emailInput`: `textContentType='emailAddress'` / `autoComplete='email'` / `returnKeyType='next'`. `welcome.otpInput`: `textContentType='oneTimeCode'` / `autoComplete='one-time-code'` / `keyboardType='number-pad'` / `returnKeyType='done'` / `onSubmitEditing` = Continue; auto-focus it when the phase flips to OTP. Focus order: email → toggles → CTA (phase 1), then OTP → CTA (phase 2). Extend the `TextField` primitive (§5) to accept these props so every field benefits.

**Animation / loading.** Email→OTP: the field swap cross-fades/rises via `FadeRise`; the Orb shifts `idle → thinking` while `busy`. Reduce-motion: Orb calm (static bloom), no field slide — `FadeRise` collapses to a `fast` cross-fade.

### 3.2 Steady-state — `app/app/(tabs)/camera.tsx` (id root `camera.screen`)

**Layout.** Full-bleed viewfinder (`CameraView.native.tsx`) on native; on web (`CameraView.tsx` is a no-op) a flat cream branded canvas fallback that reads as Shazam's branded home. Top-left: hamburger `nav.menuButton` + `voxi` wordmark (over the native feed, behind a top cream→transparent scrim with the chrome flipped to `onAccent`/white — see §2 "Chrome over the native viewfinder"). **Center: one big tappable green capture orb** (`CaptureOrb`, ~96 diameter, flat `#29AB60`) carrying the existing `camera.shutter` id. **Instruction text above the orb** (`camera.retakeHint`, `body`, centered, `text-secondary` on the web/cream canvas, white over the native feed): *"Point at one object. Fill the frame."* softening to *"Tap to identify."*

**Capture-orb glyph.** A single **1.5px white line** aperture/reticle-target mark — NOT a filled camera icon, NOT Shazam's concentric-S, NOT the violet→cyan→gold narrator gem, NOT a circle-in-circle. The flat green disc is the one large solid shape amid the pill/hairline vocabulary; keep the rectilinear Recently-Found cards and detail sheet as intentional shape counterpoint. Bump the glyph stroke (or add a 1px darker-green `#1F8A4C` inner ring) so it clears **≥3:1** non-text contrast against the disc (the white-on-green fill exception, ~2.96:1, covers the disc itself but the glyph is the only affordance and needs 3:1). The orb carries no baked-in text; the single instruction line above it is the state channel: ready → *"Tap to identify."*, busy → *"Capturing…"*. `CaptureOrb` sets a human `accessibilityLabel='Identify object'` + `accessibilityState={{busy}}` + disabled styling while busy; pressed fill `#238C4F`.

**Reticle.** Show the faint corner reticle **only when a real viewfinder is behind it** (native, camera granted). On the web/cream fallback or when camera is not granted there is no subject to frame — suppress the corner reticle (it currently renders unconditionally at `camera.tsx:143`); show nothing behind the orb, or at most a single faint concentric framing ring that reinforces (not competes with) the orb, preserving the single center of gravity.

**Narrator presence on home (decision).** Keep a *small* narrator presence so the living Guide doesn't vanish on the screen the user dwells on longest, without stealing the green orb's capture job: dock a compact narrator `Orb` (~40–48, violet-cyan-gold, `idle` breathing) beside the wordmark / top-right, and give the instruction line the Guide's voice, sourced from `useQuery(['threads']).length` (UI-only): first visit *"Show me something human-made."*, return *"Back again? Point me at something."*, post-catch *"What's next?"*. Reduce-motion: static bloom. **Keep `id={ids.processing.orb}` on the camera home's docked narrator Orb** (and on the PRIMING/DENIED narrator Orbs), preserving today's states exactly as `camera.tsx` renders them: `idle` when granted & not busy, `thinking` when busy, `listening` while priming, `uncertain` when denied. **Do NOT remove `processing.orb` from the camera route** — `camera-rnw.web.ts:36-39` asserts `d.state(ids.processing.orb).attrs.state === 'idle'` on the *camera* screen, so removing it turns the camera converge gate RED, and no replacement id is registered. The resize (~40–48) and reposition beside the wordmark are cosmetic; the id/state are what the gate reads. (The "mounted on two screens at once" worry is moot: converge mounts a single camera entry, and camera vs. processing are separate routes.)

**Bottom: "Recently Found" carousel** (`RecentlyIdentified`) peeking ~120pt.

**Recently-Found carousel.** Horizontal `ScrollView` (container `camera.recent`) of square cards (`radius.lg` 16, ~104pt, the `shadow` token), each a thread (`camera.recentItem`). Fed from `useQuery(['threads'])` — a *peek of the same query data* as Collection, latest **8 by `createdAt` desc**, not a second source.

- **Header row** (lightweight, so the growing catalog isn't hamburger-only): left = *"RECENTLY CATALOGUED"* (`overline`, `text-tertiary`, in-persona — not the flat "Recently Found"); right = a **blue** (`#3D89F5`) *"See all"* link routing to `/(tabs)/threads` (reuse `nav.threadsTab` semantics) so Collection is a first-class home entry point in addition to the drawer.
- **Card treatment (mini-ResultHero, but NO photo).** `ThreadSummary` is only `{ threadId, title, createdAt }` — there is **no** thumbnail URL and adding one is a contract change the UI-only constraint forbids. So render cream tiles (`surface-warm` `#FBF9F3`, `radius.lg`, `shadow` token) using only UI-only fields: auto-title in `name` weight (16/600, `text-primary`) plus, optionally, the `caption` timestamp demoted to `text-tertiary`. **No per-card confidence chip / band dot** — `band`, exactly like the thumbnail, is **not** on `ThreadSummary` (it exists only transiently in the per-scan `confidence_band` stream / `captureStore`), so a per-card confidence semantic would require the same forbidden contract change (add `band?`) or an 8-thread re-stream — both out of UI-only scope. `RecentlyIdentified` / `ResultHero` prop types must NOT assume the list query carries `photoUri` or `band`. (Thumbnails AND a per-card band, if ever wanted, are a separate contract PR adding `thumbUrl?` / `band?` to `ThreadSummary` — out of scope here.)
- **Tap = the canonical revisit contract**, identical to `threads.tsx` `openThread`: `reset(); setThread(item.threadId); router.push('/processing')` (re-streams the durable session → reveal/interview). **Do NOT** `router.push('/reveal')` directly — that flashes the empty "Settling on a title…" shell and bounces back through `/processing` anyway (`reveal.tsx:44`).
- **Three distinct states inside `camera.recent`** (pass `isLoading`/`isError`/`refetch` into `RecentlyIdentified` — still one query, UI-only). A loading OR errored query must never collapse to the empty ghost and falsely tell a returning collector they have zero finds:
  - **LOADING** = three 104pt skeleton tiles (`surface-sunken` `#EDEAE0` shimmer, motion row 11).
  - **ERROR** = a single cream `surface` tile *"Couldn't reach your collection."* with an inline blue *"Retry"* link calling `refetch` — never the empty ghost.
  - **EMPTY** (fetched, zero threads) = one low-emphasis dashed ghost tile, in-persona: *"Nothing catalogued yet. Point me at something."* (`text-tertiary`) that never competes with the orb.
- Mounts with an upward `FadeRise` (cap stagger ~8 tiles).

**Catch reward on return.** When the user returns to the capture home after a new identification, animate the new capture sliding into the **front** of Recently-Found with a one-time green-ring flourish so the collection visibly grows (reduce-motion: cross-fade the new tile in, no slide).

**First-run vs returning (one home, differentiated by content only — no separate first-run home).** First-capture coaching lives in `first-run.tsx`. On camera: when `threads.length === 0` show the single dashed ghost tile + the warmer first-touch instruction (*"Point at one object to begin."*); once ≥1 thread exists, show the populated carousel + the steady instruction (*"Tap to identify."*).

**Responsive.** Declare **portrait-lock** for the camera + reveal hero screens (an iOS-first capture app), OR give explicit short-viewport rules: on an iPhone SE (568pt) / short web window the carousel collapses to a single row or hides below a height threshold so it never overlaps the orb + instruction; the orb stays centered; the CTA stays reachable. Add an SE-sized and a short-web-window check to §9 verification.

**States.** permission **undetermined (priming)**, **denied**, ready, `busy`, offline, capture-error, and the web no-camera fallback — rendered in the new full-bleed canvas (not the old centered columns):
- **PRIMING (undetermined/requesting):** cream canvas, narrator `Orb` `listening` centered in the capture-orb slot, `body` *"Waking the lens…"*, **no** tappable green `CaptureOrb` yet (nothing to capture until granted).
- **DENIED:** cream canvas, `Orb` `uncertain`, a white `surface` card *"The Guide can't see."* + green `camera.openSettings` pill (`camera.permissionDeniedBanner` preserved). Keep the hamburger + wordmark present so the drawer stays reachable from denied.
- **OFFLINE / capture-error:** pin `global.offlineBanner` as a full-width strip directly under the hamburger/wordmark row (above the viewfinder); the orb stays tappable. On capture failure the `CaptureOrb` returns from `busy` to idle (never stuck on "Capturing…") and re-enables; if offline, swap `camera.retakeHint` to the offline retry copy (*"We're offline — your last capture will retry…"*); if online, a calm in-persona line (*"That didn't take — tap to try again."*). 402 still routes to `/paywall`.
- **WEB:** no-camera cream fallback canvas (below).

**Web no-camera fallback (the surface the harness + web visitors actually render).** The identical home skeleton minus the live feed: hamburger + wordmark top-left, `CaptureOrb` centered, instruction above, Recently-Found below, and a faint `hairline`-stroked corner reticle framing the center where the viewfinder would sit. **Explicitly forbid** any decorative blob, gradient wash, or oversized hero logo — the cream canvas + reticle IS the branding, so it reads as the same branded home, not an empty placeholder.

**Shared secondary-state layout (consistency with the off-center hero screens).** Denied-permission, offline, and capture-error states use one shared editorial layout: narrator Orb docked small at top, text **LEFT-aligned at the screen gutter** (not dead-centered), the primary action as the 52pt green pill pinned low — matching the login/reveal alignment language. The one deliberate centered exception is the in-flight processing screen (orb + PulseRings). This same layout applies to the cold reveal and the threads empty/error states.

**Animation / loading.** Capture orb idle "breathing" (scale 1→1.04 + bloom opacity 0.22→0.30, full period `orbIdle` 4200ms); press `scale→0.94` on `onPressIn` (`fast`), spring rebound on release. On press: orb pulses green once, `busy` disables it (label "Capturing…"), then `api.createThread` → `router.push('/processing')`. A pseudo-shared-element hand-off (orb shrinks + drifts toward center over `base`) into processing. Reduce-motion: freeze breathing to a static bloom; keep the `scale→0.96` press confirm.

### 3.3 Processing — `app/app/processing.tsx` (id root `processing.screen`)

**Layout.** Full-bleed cream. **X cancel top-right** (`processing.cancel`): aborts the stream and `router.replace('/(tabs)/camera')` — a *navigational* line glyph in `text-primary`, never terracotta; give it a **44pt hit box** around the glyph and `accessibilityLabel='Cancel'`. **Center: the just-captured photo behind the Orb** — render `captureStore.photoUri` (already used on reveal) as a dimmed, cream-scrimmed square (`radius.lg`) *beneath* the narrator Orb, so the wait feels like the Guide is examining **your** object, not buffering an empty void. **Over it, the narrator Orb (`processing.orb`, carries `orb.state`) inside `PulseRings`** — 3 concentric rings radiating outward, green `#29AB60` at 0.35 / 0.12 / 0.05 opacity, staggered ~1400ms, expanding + fading (thin hairline strokes, no glow) so the photo reads as being scanned. On the CONFIDENT settle the photo brightens as rings collapse to the bloom, bridging directly into the reveal hero's Ken-Burns (§3.4). Reduce-motion: static dimmed photo, no scan sweep. **Staged status text below** (`processing.loadingLine`, `headline` `text-secondary`, optional overline stage-eyebrow) — capped at **max 2 text rows at any time** (one dominant status line + an optional small overline eyebrow; the `longWaitAck` *replaces* the secondary/subtext row rather than adding a third line; the three-dot ellipsis stays inline on the status line, not its own row) — choreographed to real stream events (mirrors Shazam's "Listening → Searching → Expanding search"):

- t0 `thinking`: "Consulting the Guide…"
- `partial_id`: "Looks like {title}. Confirming…" (rings tighten)
- ~9s `longWaitAck` (`processing.longWaitAck`): "Still here. Some objects are coy." + animated three-dot ellipsis.
- `token` (narration streaming) `speaking`: the rings **drop the outward pulse and switch to a slow synced "breathing"** while tokens stream (they must not freeze oddly nor keep radiating as if still searching).
- Terminal settle (~450ms, observable): CONFIDENT "I've got it: {title}." → `/reveal`; PROBABLE announces refinement explicitly → partial/`/reveal`; UNKNOWN "I don't know this one — yet. ## 4. Motion & loading states

**Governing rule (resolves the motion-library conflict).** The web converge harness (`e2e/web/converge/harness.ts`) aliases only `react-native-web`, `expo-router`, `expo-image`, `expo-constants`, `react-native-safe-area-context`, `@tanstack/react-query`, `zustand`, `react`/`react-dom`. It does **not** alias `react-native-reanimated`, `react-native-gesture-handler`, `@shopify/react-native-skia`, or `expo-haptics`. **The converge esbuild bundle statically resolves EVERY `require()`/`import()` string literal regardless of any surrounding `Platform.OS !== 'web'` guard** — `Platform.OS` is a runtime react-native-web value, not a build-time define, so there is NO dead-code elimination and even a lazy `import('react-native-reanimated')` is inlined by `format:'iife'` and resolved. So a bare guard is a **build-time no-op**: it prevents runtime *execution* on web (the throwing shim is never invoked), but the harness **alias** is what actually keeps the module out of the bundle. A native-only lib is web-safe ONLY when it is aliased to a bundleable stub (as `shims/absent-native.tsx`'s header states and the 6 existing native modules already do). Consequently: **everything on the E2E-asserted / converge path uses RN `Animated` (the proven `FadeRise` seam that renders identically on web + native and honors `reduceMotion`).** Reanimated / gesture-handler / Skia are permitted ONLY as native-only progressive enhancements — each kept behind a `Platform.OS !== 'web'` guard for runtime safety AND **aliased in the harness** (foundation step 1: `react-native-reanimated`, `react-native-gesture-handler`, `@shopify/react-native-skia` → `shims/absent-native.tsx`) so the bundle still builds. Haptics (`expo-haptics`) are native-only, always guarded `Platform.OS !== 'web'`, AND likewise aliased to a web no-op shim (it too fails to bundle via expo-modules-core's `TurboModuleRegistry` import, so the guard alone does not save the build). Both the guard and the alias are required; the guard alone is insufficient. ## 7. Constraints honored

- **Web/native parity.** Every animation on the converge-asserted path uses RN `Animated` (`FadeRise` seam) — all render under react-native-web. No shared screen imports `react-native-reanimated` / `react-native-gesture-handler` / `@shopify/react-native-skia` / `react-native-svg` (its native fabric build imports `codegenNativeComponent`, which RNW does not provide) at module top level. **Critically, a `Platform.OS !== 'web'` guard alone does NOT keep a lib out of the converge bundle:** the esbuild converge build statically resolves every `require()`/`import()` string literal regardless of any surrounding runtime guard — `Platform.OS` is a runtime react-native-web value, not a build-time define, so there is no dead-code elimination. A native-only lib is web-safe ONLY when the harness **aliases** it to a bundleable stub (as `shims/absent-native.tsx`'s header states and the 6 existing native modules already do). The `Platform.OS` guard governs runtime *execution* (skips the require on web so the throwing shim is never invoked); the harness *alias* is what lets the build succeed — **both are required, the guard alone is insufficient.** Therefore `react-native-reanimated`, `react-native-gesture-handler`, `@shopify/react-native-skia`, and `expo-haptics` must be **added to the harness alias map** (`harness.ts` alongside the existing 6) — reanimated/gesture-handler/skia → `shims/absent-native.tsx`, `expo-haptics` → a small web no-op shim (it also fails to bundle via expo-modules-core's `TurboModuleRegistry` import, so it must be aliased, not merely guarded) — as a foundation-step prerequisite (see §5 step 1 / §8). Prefer funnelling all native-only motion + haptics through a single seam module (e.g. `app/src/lib/nativeMotion.ts`) that does the guarded requires, and alias THAT one module, forbidding direct reanimated/gesture-handler/skia/expo-haptics imports in any converge-reachable component. `CameraView` is a no-op on web, so steady-state renders the cream branded fallback in the harness.
- **Reduce-motion.** Every effect reads `useTheme().reduceMotion`; §4 gives each a concrete degradation (cross-fade, static rings/bloom, near-instant tile fades) — the orb is never hidden, nothing becomes unreadable. The Settings `reduceMotion` toggle still overrides the OS bridge.
- **UI-only.** No changes to `services/voxi-api` (BFF), the NDJSON/events taxonomy in `packages/shared/src/events.ts`, the api client shapes, metering/charging, or ACLs. The `capture → api.createThread → api.streamThread → reveal` flow, terminal-outcome routing (CONFIDENT/PROBABLE/UNKNOWN), `?startIndex=` reconnection, and 402→`/paywall` behavior are all preserved. No E2E assertion or honesty-gate stage is weakened.
- **Buildable.** Expo / expo-router only; no new native modules requiring a custom dev client beyond what's installed (reanimated 4.5, gesture-handler ~2.32, skia 2.6, svg 15 already present; only ever used native-guarded). Drawer is a custom `DrawerHost` (RN `Animated`), deliberately **not** `@react-navigation/drawer` (which would drag in gesture-handler + reanimated and kill converge parity).

## Unresolved design decisions

Genuine forks deferred to implementation. Each has a **recommended default** already folded into the spec above; a builder may overturn it, but must pick consciously.

1. **Reveal: one green primary — pill vs. action orb.** Two green discs (pill + bottom-right orb) both read as THE primary. **Default (folded in):** the 52pt band-labelled **pill is the sole primary**; the bottom-right green orb is a subordinate **icon-only narration play/pause toggle** (`reveal.playNarration`, ~56pt, smaller than the pill). Alternative: make the orb itself `reveal.primaryAction` and drop the pill (more Shazam-literal, but weaker for a text-labelled band CTA). Either way, exactly one element is primary.
2. **Drawer navigation model.** **Default (folded in):** the drawer hosts ALL primary nav Shazam-style — add a top **Capture/Home** row (`drawer.home`) and navigate rows with `router.navigate` (no stacking); pushed `threads`/`settings` also get a `nav.close` back-chevron. Alternative: keep the hamburger only on camera and rely solely on back-chevrons elsewhere (lighter drawer, but the home surface is one hop further away).
3. **Narrator Orb on the capture home.** **Default (folded in):** a *small* docked idle narrator Orb beside the wordmark keeps the "living Guide" present on the most-dwelt screen; it **keeps `id={ids.processing.orb}` in state `idle`**, which is exactly what `camera-rnw.web.ts` asserts — no id is removed. Alternative: the Guide "appears when summoned" and the idle home shows no narrator Orb (cleaner single-center-of-gravity, but the character goes mute at the hub) — **note this alternative is NOT converge-safe**: `camera-rnw.web.ts:36-39` requires a `processing.orb` in state `idle` on the camera screen, so removing the home Orb would fail the gate unless the frozen runner is edited (out of scope). The retain-the-Orb default is therefore the only option that passes the gate without touching the runner.
4. **Reveal title legibility technique.** **Default (folded in):** an **opaque cream placard band** under the title (gradient-free, AAA on any photo). Alternative: a tuned minimum-opacity flat cream scrim + text-shadow token over the photo (more Shazam-like full-bleed, but risks AA on worst-case photos and needs per-photo tuning).
5. **Recently-Found thumbnails.** **Default (folded in):** **title-only cream cards** (title + optional timestamp), because `ThreadSummary` carries neither a thumbnail nor a `band` — both would be contract changes the UI-only constraint forbids (`band` lives only in the transient per-scan `confidence_band` stream / `captureStore`). Alternative: a separate, explicitly-scoped contract PR adding `thumbUrl?` **and/or `band?`** to `ThreadSummary` to enable photo mini-heroes and/or a per-card confidence dot later.
6. **Reveal narration: READY-on-mount vs. live-streaming.** `processing.tsx` drains the whole stream (breaks on `done`) before routing, so `reveal.tsx` mounts with `whatItIs` already complete and imports no api client. **Default (folded in, UI-only):** reveal is **READY-on-mount** — no progressive token fill, no shimmer-until-`done`, no "disabled while text streams" orb sub-state.
